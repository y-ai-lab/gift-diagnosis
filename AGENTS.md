## 1. このリポジトリの目的

このリポジトリは、プレゼント選びが苦手な人の迷いを減らすための無料診断サイトです。

対象サイト：

- https://y-ai-lab.github.io/gift-diagnosis/

このサイトの目的は、単に「おすすめジャンル」を表示することではありません。

診断を受けた人が、

- 何系を選べばいいか
- 第一候補と第二候補は何か
- 何を避ければいいか
- 迷ったら何を選べばいいか
- どの検索ワードを使えばいいか
- 次にどこで探せばいいか
- 渡すときに何と言えばいいか

まで分かり、プレゼント選びを前に進められる状態を作ることが目的です。

---

## 2. 対象ユーザー

主な対象は以下の人です。

- プレゼント選びに毎回時間がかかる
- 候補が多いと決められない
- センスがないと思われたくない
- 無難すぎるのも、外すのも怖い
- Amazonや楽天を長時間見て疲れる
- 相手の好みを深く知らない
- 難しい説明や抽象的な言葉で止まりやすい
- スマホで短時間に答えを出したい

初心者や優柔不断な人を前提に、難しい判断をユーザーへ戻さないことを優先してください。

---

## 3. プロダクトの基本方針

### 3.1 診断の役割

この診断は、商品を大量に見せるカタログではありません。

大手ECサイトやギフト専門店と商品数で競わず、以下に特化します。

> プレゼント選びが苦手な人の迷いを減らし、候補を具体的な検索行動まで絞る。

### 3.2 結果ページで必ず目指す状態

結果ページでは、最低限以下を分かりやすく表示してください。

- おすすめの方向性
- 第一候補
- 第二候補
- 避けた方がいいもの
- 迷ったときの無難な選び方
- そのまま使える具体的な検索ワード
- 検索ワードのコピー機能
- Amazon・楽天・Yahooショッピング等の検索結果への導線
- 渡すときの一言
- 次にやること

結果を見たユーザーに「結局、自分でまた一から探すのか」と思わせないことが重要です。

### 3.3 誠実さ

- 喜ばれることを保証しない
- 完璧な正解を断定しない
- 診断結果は選び方の目安として示す
- アフィリエイトリンクを使う場合はPR表記を明記する
- アフィリエイトの都合で診断結果を歪めない
- 有料登録や会員登録を診断の条件にしない
- ユーザーの回答を外部送信しない

## 4. 文章ルール

### 4.1 基本

- 中学生でも読める日本語を使う
- 1文を長くしすぎない
- 抽象語だけで終わらせない
- 機能説明より、次に何ができるかを伝える
- きれいすぎる営業文より、自然で誠実な文章を優先する
- 成功や喜ばれる結果を保証する表現を避ける

### 4.2 避ける表現

以下のような抽象的な表現だけで終わらせないでください。

- おしゃれなギフト
- 実用的なもの
- 相手に合うもの
- 外しにくい方向性
- おすすめジャンル
- 次に探してみましょう

必ず、検索や購入行動に移せる粒度へ具体化してください。

悪い例：

> おしゃれなお菓子ギフト

良い例：

> 3,000円前後 個包装 焼き菓子 ギフト 職場 お礼

### 4.3 専門用語

次のような言葉は、画面上ではなるべく避けてください。

- CTA
- MVP
- LP
- コンバージョン
- パラメータ
- アフィリエイト導線

必要な場合は、ユーザー向け画面では自然な日本語に置き換えてください。

例：

- CTA → 次にやること
- MVP → まず動く試作品
- LP → 商品説明ページ
- コンバージョン → 次の行動につながった数

## 5. 診断質問・ロジックのルール

### 5.1 勝手に変更しないもの

以下は、事前の計画確認なしに変更しないでください。

- 質問数
- 質問の順番
- 選択肢
- 配点
- 結果判定ロジック
- 結果パターン
- 回答データの形式
- URLパラメータ形式
- 既存の結果復元仕様

### 5.2 質問追加の判断

質問を増やす場合は、次の条件を満たす必要があります。

- 結果の具体性が明確に上がる
- 検索ワードの精度に反映できる
- 回答負担より改善効果が大きい
- スマホで1〜2分程度の軽さを大きく損なわない
- 追加した回答が結果文に実際に反映される

追加候補として優先度が高いもの：

- 相手の年代

条件付き候補：

- 地域  
  体験ギフトや店舗検索を結果に含める場合のみ検討する

追加前に、既存回答だけで検索ワードを具体化できないか確認してください。

### 5.3 複数選択

「避けたいもの」は複数当てはまる可能性があります。

複数選択化する場合は、以下を確認してください。

- 複数回答が結果にすべて反映される
- 矛盾する条件を安全に処理できる
- 未選択でも進められるかが明確
- スマホで選択状態が見やすい

## 6. スマホ優先UIルール

このサイトはスマホ利用を最優先にします。

必須確認幅：

- 375px
- 430px
- デスクトップ幅

必須ルール：

- 横スクロールを発生させない
- 主要ボタンの高さは48px以上
- ボタンや選択肢の間隔を十分に取る
- 文字切れを起こさない
- 下部固定ボタンが本文に重ならない
- 質問遷移時に適切な位置へスクロールする
- 選択状態を色・枠・チェック等で明確にする
- コピー完了時に「コピーしました」と表示する
- 外部リンクは押した先が分かる文言にする
- 結果ページは結論を先に表示する
- 長い補足は折りたたみを使う
- 第一候補と次の行動をスクロール上部で確認できるようにする

### 次へ・戻る

- スクロールしないと押せない状態を避ける
- 固定表示する場合は本文や端末の安全領域に重ならないこと
- 「戻る」は選択肢と誤認しない見た目にする
- 単一選択で即遷移する場合も、誤操作時に戻れること

## 7. 検索ワードと外部検索のルール

検索ワードは、抽象語の羅列ではなく、実際のEC検索に使える形にしてください。

基本構成：

> 予算 + 具体的な品名・ジャンル + 用途 + 相手との関係 + 安全条件

例：

> 3000円 個包装 焼き菓子 ギフト 職場 お礼

検索ワードには、可能な範囲で以下を含めます。

- 予算
- 品名または具体的ジャンル
- 用途
- 相手との関係
- 年代
- 個包装、常温、消えもの等の条件

外部検索リンクは、モールのトップではなく検索結果へ直接移動させてください。

外部リンクを追加する場合は以下を確認してください。

- 検索語が正しくURLエンコードされる
- 新しいタブで開く
- `noopener noreferrer` 等の安全な設定を使う
- リンク切れがない
- PR表記が必要なリンクには明記する
- 予算の上限・下限をURLへ反映できる場合は反映する

## 8. 結果ページの情報順

原則として、以下の順番を優先してください。

1. 診断結果の結論
2. 第一候補
3. 選んだ理由
4. 第二候補
5. 迷ったときの無難な選び方
6. 避けた方がいいものと理由
7. 検索ワード
8. 検索ワードをコピー
9. 商品を探すボタン
10. 渡すときの一言
11. 次にやること
12. 今回の回答条件
13. 回答を変えて再診断
14. 運営者・X・note・免責

ユーザーの入力内容や長い説明を、結論より先に置かないでください。

## 10. 技術的制約

- GitHub Pagesで動く静的構成を維持する
- 既存のビルド方式を勝手に変更しない
- 外部APIを勝手に追加しない
- AI APIで診断結果を生成しない
- 個人情報・決済・ログイン機能を追加しない
- 既存URLを変更しない
- サブディレクトリ配信で相対パスが壊れないようにする
- 依存パッケージを増やす場合は事前に理由を示す
- 関係ない大規模リファクタリングをしない
- ユーザーの未コミット変更を上書きしない

## 11. Gitルール

作業開始前：

- `git status` を確認する
- 現在のブランチを確認する
- 未コミット変更を確認する
- ユーザーの変更を消さない

作業中：

- 1回1目的を基本にする
- 関係ない整形を混ぜない
- 大規模変更はPlan作成後に行う

作業終了時：

- 差分を自己レビューする
- テストを実行する
- 画面確認を行う
- `git status` を報告する
- 指示がない限りcommitしない
- 指示がない限りpushしない
- 指示がない限り公開環境へデプロイしない

## 12. Codexモデルの選び方

Codexへの依頼では、必ず推奨モデル、推論強度、選定理由を示してください。

### Luna

対象：

- 文言変更
- リンク差し替え
- 軽微なCSS
- 小さなテスト追加

推論強度：

- 低〜中

### Terra

対象：

- 検索ワードコピー機能
- EC検索リンク
- スマホUI改善
- 数ファイルにまたがる通常改修
- 結果ページの構造整理

推論強度：

- 中

### Sol

対象：

- 質問追加
- 診断ロジック変更
- URLパラメータ化
- 複数選択化
- 原因不明の不具合
- 大規模な結果データ刷新
- 100項目レビューの計画化

推論強度：

- 高

`max` は失敗コストが高い重要改修だけに使用します。  
`ultra` は複数エージェントが必要な大規模調査・改修だけに限定します。

## 13. タスクの進め方

### 小さな修正

流れ：

> 調査 → 直接実装 → テスト → 報告

### 通常改修

流れ：

> 調査 → 短い計画 → 実装 → テスト → ブラウザ確認 → 報告

### 複雑な改修

流れ：

> Planモード → 計画確認 → 第1段階のみ実装 → レビュー

100項目レビューを一度にすべて実装しないでください。

計画は大きく整理しても構いませんが、実装は小さく分けます。

## 14. 必須テスト

変更内容に応じて、最低限以下を確認してください。

### 診断フロー

- 最初から最後まで完走できる
- 戻る操作ができる
- 回答を変更すると結果が再計算される
- 各質問の選択状態が正しく表示される
- 複数選択がある場合、複数回答が保持される

### 結果

- 第一候補・第二候補が表示される
- 避けたい条件が結果へ反映される
- 予算が検索ワードへ反映される
- 年代質問を追加した場合、年代が結果へ反映される
- 検索ワードがコピーできる
- コピー成功表示が出る
- EC検索リンクが正しい検索結果を開く
- 渡すときの一言が表示される
- 結果の再表示・復元機能がある場合、正しく復元される

### スマホ

- 375px
- 430px
- 横スクロールなし
- 固定ボタンの重なりなし
- 文字切れなし
- タップ領域48px以上
- 長い表現が読みやすい

### 技術

- ビルド成功
- 既存テスト成功
- 追加テスト成功
- コンソールエラーなし
- ネットワークエラーなし
- 外部リンク切れなし
- GitHub Pagesのパスで動作

## 15. 完了条件

コードを書いただけでは完了ではありません。

以下を満たした時点で完了です。

- 依頼された目的を満たしている
- 必須要件を満たしている
- 変更禁止事項を守っている
- 既存診断フローを壊していない
- スマホ実機相当で確認している
- テストが通っている
- コンソールエラーがない
- 外部リンクが動く
- 変更内容を自己レビューしている
- 未解決点を正直に報告している
